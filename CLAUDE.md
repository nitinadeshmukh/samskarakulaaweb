# CLAUDE.md — samskarakulaaweb

Marketing site for **Samskara Kula**, deployed at `https://samskarakula.com`. Built 2026-07-06, split-adjacent to the `samskarakulaaapi` (backend) and `samskarakulaaapp` (mobile apps) repos.

## Domain terms (use verbatim, see full glossary in the other two repos)

Samskara, Shodasha Samskaras, Gotra, Veda/Shakha, Kula Devata, Muhurta, Panchang/Ephemeris, Pandit/Purohit, Agamic tradition, Samagri, Havan/Homam. Do not paraphrase these into generic marketing-speak in copy — they're precise, and getting them wrong reads as inauthentic to the target audience.

## Tech stack

- Vite + React + TypeScript + Tailwind CSS, static build output (`dist/`).
- No backend of its own — this is a static marketing site. It intentionally does NOT talk to the Supabase backend (`samskarakulaaapi`) or reuse any app auth/data.
- Branding: genealogy-constellation mark (`src/components/Logo.tsx`), blue/navy (`kula-navy` `#0E4F9E`) + gold/amber (`kula-gold`/`kula-amber`) palette — matches `samskarakulaaapp`'s customer-app icon and UI accent color.

## Analytics (2026-07-08, same day) — the one deliberate exception to "no backend"

Per direct user request ("We will have to study user behaviours when on our site to provide them services, work on it so we can cater better"). Confirmed with the user first: track website **and** customer-app (not priest-app), self-hosted custom pipeline in Supabase (not a third-party vendor — data-residency/DPDP reasons per root `CLAUDE.md` §7 rule 6), consumed via an admin-app dashboard (not yet built — see `samskarakulaaapp`'s task list).

This is a **narrowly scoped exception** to "no backend of its own" above — the site now talks to Supabase for exactly one thing: write-only inserts into `analytics_events` (`samskarakulaaapi` migration 0017). Nothing else about the site's architecture changed; it does not read from Supabase, does not use auth, does not touch any other table.

- **`src/lib/analytics.ts`** — `trackEvent(eventName, properties)`, writes `{source: 'website', session_id, event_name, event_properties}`. `session_id` is a `crypto.randomUUID()` generated once and persisted in `localStorage` (`sk_session_id`) — anonymous, first-party, not a fingerprint, not shared cross-site. Only ever captures UI-navigation/interest signals (page view, section scrolled into view, button clicked) — **never** anything a visitor typed into a form (the waitlist email itself is not sent as an event property).
- **`src/hooks/useSectionView.ts`** — IntersectionObserver-based, fires a `section_viewed` event once per section per page load (threshold 0.3, disconnects after first fire). Deliberately separate from `Reveal.tsx` (that component fires per-item for the fade-up animation, too granular/noisy for section-level analytics).
- Wired: `page_view` (`App.tsx`, on mount), `section_viewed` × 6 sections (Problem, Features, HowItWorks, RatingsReviews, Priests, ServicesShowcase), `cta_clicked` on both Nav CTAs (desktop + mobile) and both Hero CTAs, `waitlist_submitted` on the waitlist form's actual submit handler.
- **Env**: `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` in `.env` (gitignored — this repo never had a `.env`/`@supabase/supabase-js` dependency before this pass, both newly added). Same anon key value already used publicly by the other 3 apps — reusing it is not a secret-handling issue, since the anon key is client-public by design and protected by RLS, not secrecy.
- **`src/vite-env.d.ts`** (new) — `/// <reference types="vite/client" />`. Needed once `import.meta.env` was actually used for the first time in this repo; without it, `tsc -b` fails with `Property 'env' does not exist on type 'ImportMeta'`. (Separately, this repo's `npm run typecheck` script itself has a pre-existing, unrelated bug — `tsc -b --noEmit` errors with `TS6310: ... may not disable emit` because `-b`/build-mode and a CLI `--noEmit` override conflict; this predates this session, reproduces on a clean `git stash`, and isn't part of the analytics work. Use plain `npx tsc -b` to actually typecheck this repo until the script is fixed.)

**A real bug was found and fixed during live verification, not just typechecked**: the original `trackEvent` implementation used `void supabase.from('analytics_events').insert({...})` as a "fire and forget" pattern. This looks reasonable for a normal `Promise` but is wrong for `@supabase/postgrest-js`'s query builder — it's a **lazy thenable**: the actual `fetch()` only happens when the builder is awaited or `.then()`'d. `void builder` alone evaluates the expression and discards it *without ever calling `.then()`*, so the request silently never fired. No analytics event had ever actually reached the database despite the code typechecking and looking correct. Confirmed via live network inspection (zero POST requests to `analytics_events` despite firing `trackEvent` multiple ways) before diagnosing the cause. Fixed by chaining `.then(() => {}, () => {})` to actually trigger execution while still swallowing both success and error paths silently (per the "should never break the page" comment already there). **Re-check this pattern if `trackEvent` or any other bare `supabase.from(...).insert(...)` call is ever written without an `await`/`.then()`/`.select()` — the lazy-thenable footgun applies to any postgrest-js builder, not just this one.**
- Live-verified end-to-end after the fix: reloading the page produces a real `page_view` row, clicking a "Join the Waitlist" CTA produces a real `cta_clicked` row with `{cta: "join-waitlist", location: "nav"}`, both confirmed via direct `psql` query against the production table (test rows cleaned up by exact ID afterward, not by pattern-match delete).
- The `analytics_events` RLS insert policy (`family_account_id is null or family_account_id in (select ... from family_accounts ...)`) is correct and was independently confirmed to work as designed once a `RETURNING`/`Prefer: return=representation` isn't attached to the request — a stray manual test that added that header hit the (also-correct, admin-only) `analytics_events_admin_read` SELECT policy instead, which was the source of an initial false alarm during debugging, not a real RLS bug. `supabase-js`'s `.insert()` does not request a representation unless `.select()` is chained, so this doesn't affect real usage.

## Production outage (2026-07-08) — `analytics.ts` crash on first real CI deploy

The accumulated redesign work (Justinmind hero, SacredSymbols/ServicesShowcase/RatingsReviews, the Om-medallion logo swap) had built up locally across several sessions without ever being committed, since a Vite build needs the whole tree — pushing the logo change meant pushing everything at once. That push (`2facffb`) triggered this repo's `deploy.yml` (no PR gate — every push to `main` auto-builds and rsyncs straight to production) and broke the live site: a completely blank page.

**Root cause**: `src/lib/analytics.ts`'s `createClient(import.meta.env.VITE_SUPABASE_URL, ...)` call threw `Error: supabaseUrl is required.` synchronously at module load (this module is imported from `App.tsx`), because the GitHub Actions build environment had never had `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` set — this was the first time this repo's CI had ever actually run a build; previously it had only ever been built locally, where a dev `.env` supplied the vars. The bug had existed since analytics was first added but was never exercised until this deploy.

**Fix** (`02bd6bf`): guarded the client construction (`supabaseUrl && supabaseAnonKey ? createClient(...) : null`) plus an early return in `trackEvent` if `supabase` is null, so a misconfigured env now degrades to "analytics silently doesn't fire" instead of "the whole site is down." Also added the actual public anon key + URL as literal `env:` values directly in `.github/workflows/deploy.yml`'s build step (not GitHub secrets — this is the same public, RLS-protected anon key already shipped in all 3 Expo apps' bundles, not a secret in need of protecting). Confirmed fixed live: page renders correctly, no console errors, new logo shows in nav/footer.

**Standing implication**: any code added to a module that runs at import time (not just inside a component/function) must not throw on a missing env var — this repo's CI has no staging step, so a bug like this goes live instantly on the next push to `main`.

## GPT/AI-crawler optimization (2026-07-10)

Per direct user request ("The website needs to be GPT Optimized"). Two distinct problems, both addressed:

**1. This is a client-rendered SPA — crawlers that don't run JS saw nothing.** `vite build` alone ships `dist/index.html` with an empty `<div id="root">`; the real content only appears after React executes in a browser. GPTBot, ClaudeBot, and most other AI/LLM crawlers as of 2026 do **not** execute JavaScript, so they'd have indexed a blank page. Fixed with a build-time prerender step, not metadata alone:

- **`src/entry-server.tsx`** (new) — thin SSR entry: `render(path) => renderToString(<App path={path} />)`. Only ever imported by the prerender script via Vite's `ssrLoadModule`, never shipped to the browser.
- **`src/App.tsx`** — now takes an explicit `path` prop instead of reading `window.location.pathname` directly (there's no `window` in the Node prerender process). Every `window`/`document`/`localStorage` reference elsewhere in this app (`Nav.tsx`'s scroll listener, `Reveal.tsx`/`useSectionView.ts`'s `IntersectionObserver`, `analytics.ts`'s `getSessionId`) was already safely confined inside `useEffect`/deferred function bodies, which never run during `renderToString` — no other refactor was needed for SSR-safety.
- **`scripts/prerender.js`** (new) — runs after `vite build` (see the `build` script in `package.json`). For each of the 5 routes (`/` + the 4 `/legal/*` pages), renders `App` to a real HTML string and writes it to `dist/<route>/index.html`, replacing the empty root div. Per-page `__TITLE__`/`__DESCRIPTION__`/`__CANONICAL__` tokens in `index.html` (underscore-wrapped, not `%...%`, since Vite's own HTML env-var substitution greedily parses anything percent-wrapped even inside comments — this broke the build once and was root-caused, not worked around) get filled in per route too.
- **`src/main.tsx`** — now checks `document.getElementById('root').hasChildNodes()`: if true (the prerendered-production case), `hydrateRoot`s onto the existing markup; if false (plain `vite`/`vite preview` without the prerender step, i.e. local dev), falls back to a plain `createRoot().render()`. Verified live: build+prerender+`vite preview`, hydration is clean (no console mismatch warnings), and a FAQ accordion click still works post-hydration, confirming interactivity survived.

Fixed in `src/lib/analytics.ts` by also gating the client on `typeof window !== 'undefined'` — analytics never needs a client outside a browser anyway (`trackEvent` only ever runs from a `useEffect`), so this is the correct fix, not "bump CI to Node 22 and hope nothing else hits this."
